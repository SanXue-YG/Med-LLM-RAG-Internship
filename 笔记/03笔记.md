## Q1：分割的对象是 title+abstract 还是 body？

### 问题

任务要求中说"将每篇文献的标题和摘要，解析并分割"，但第二阶段策略配置中提到了 body 的分割参数。03 任务到底要分割什么？

### 答案

**只分割 title + abstract，不涉及 body。**

依据来自 **`03 文档解析与分割/任务.txt`（老师任务书）第 4 行**：

> "根据上周制定的报告中的策略，将每篇文献的**标题和摘要**，解析并分割成适合向量化和检索的文本单元。"

这是老师的明确要求，不是我们自己的规划。

### 与第二阶段的关系

第二阶段 `chunk_strategy_config.json`（我们自己的规划文件）中定义了两套参数：

| 参数 | 值 | 用途 |
|------|-----|------|
| `retrieval_unit` | `"title+abstract"` | **本次任务的分割对象** |
| `retrieval_chunk_size` | 400 | title+abstract 的分块大小 |
| `retrieval_chunk_overlap` | 80 | title+abstract 的重叠 |
| `body_chunk_size` | 512 | body 的分块大小（**本次不用**） |
| `body_chunk_overlap` | 80 | body 的重叠（**本次不用**） |

策略注释也写明：

> "body 一律 sliding_window，**与摘要分开**"

这意味着 body 分割是**独立的后续任务**，不在 03 阶段范围内。

### 为什么不分割 body？

1. **检索场景定位**：RAG 检索通常用 title+abstract 作为检索单元（语义密集、长度适中）
2. **第二阶段已验证**：85% 的 title+abstract ≤ 512 tokens，符合 embedding 模型限制
3. **body 用途不同**：body 通常用于"检索后展示详情"或"知识抽取"，是下游任务

## Q2：任务.txt 第 8-10 行是什么意思？

### 原文

```
1环境准备与数据加载
    dataframe转换
    基础清洗
    为每篇文献生成唯一标识id（如果原始数据没有合适id）（pmid）
```

### 逐行解析

| 行 | 内容 | 含义 | 与第二阶段的对应 |
|----|------|------|-----------------|
| **dataframe转换** | 将原始数据加载为 pandas DataFrame | 第二阶段的 slim JSONL 可直接用 `pd.read_json(..., lines=True)` 读取 |
| **基础清洗** | 去除空值、异常字符、标准化格式 | 第二阶段已做：无 abstract 的记录已被过滤（`skip_no_abstract=True`） |
| **生成唯一标识id** | 如果原始数据没有合适 id，需要生成 | 第二阶段的 `pmcid` 字段**已经是唯一标识**，无需额外生成 |

### 关键结论

**第二阶段已完成这些前置工作**：

```
第二阶段 slim JSONL 结构:
{
  "pmcid": "PMC12345678",     ← 唯一标识（对应 doc_id）
  "pmid": "12345678",         ← 备用标识
  "title": "...",             ← 待分割
  "abstract": "...",          ← 待分割
  "journal": "...",
  "pub_year": 2024,
  "pub_date": "2024-01-15",
  "n_chars_abstract": 1200,
  "n_chars_body": 8500
}
```

所以 03 阶段的"环境准备与数据加载"可以简化为：
1. 读取 `oa_comm_slim.jsonl`
2. 字段映射：`pmcid → doc_id`，`title → source_title`
3. 拼接：`text = title + "\n\n" + abstract`

## Q4：为什么可以跳过 XML 直接用 JSONL？

### 问题

任务说"输入 PubMed 数据集"，是否需要回到原始 XML？

### 答案

**不需要**。第二阶段的 slim JSONL 已经是"解析后的结构化数据"：

| 原始数据 | 第二阶段做了什么 | 03 阶段需要 |
|---------|----------------|------------|
| 500万 XML 文件 | 解析提取 title, abstract 等字段 | ✅ 已有 |
| 无 abstract 记录 | 过滤并记录到 skipped_log | ✅ 已处理 |
| 字段清洗 | XPath 修正、格式标准化 | ✅ 已完成 |

所以 03 阶段的"解析"实际上是**字段映射 + 拼接**，而非重新解析 XML。
